왜 일관되게 줄어드는가

  • HTML은 구조를 위한 태그가 본문보다 많다 (<div>·<span>·인라인 스타일·attr 등)

1.2 파싱 정확도

구조가 있는 평문이 가장 안전하다

서식 · 메타데이터가 섞인 바이너리보다, 의미적 구조가 살아 있는 평문이 LLM의 일관성을 끌어올린다. 작업 유형에 따라 3가지 포맷이 명확하게 갈린다.

구조 작업 Markdown

  • 문서 Q&A · 헤딩 · 코드블록 인식이 필요한 경우
  • 의미적 계층(H1~H4)이 자연스럽게 청크 경계로 활용됨 (LlamaIndex MarkdownNodeParser)
  • RAG 인입 표준 ― 검색 정확도 향상의 가장 큰 변수

사실 추출 · 요약 평문

  • 단순 텍스트 처리, 키워드 추출, 짧은 요약
  • 구조 메타데이터가 오히려 노이즈로 작용
  • 토큰 비용 최소화가 우선인 워크로드

시각적 렌더링 HTML

  • 인터랙티브 UI · 차트 · 폼이 필요한 산출물
  • Claude Artifacts · ChatGPT Canvas 영역

1.3 OpenAI 공식 가이드

OpenAI ― "Markdown으로 시작하라"

GPT-4.1과 GPT-5의 공식 프롬프트 가이드 모두 Markdown을 구조화의 출발점으로 명시한다. 추론·코딩 워크로드 전반에 걸친 일관된 권고다.

GPT-4.1 Prompting Guide developers.openai.com

  • "We recommend starting here" — Markdown을 프롬프트 구조화의 출발점으로 명시
  • 주요 섹션·서브섹션은 Markdown 헤더로 (H1~H4+ 깊은 계층까지)
  • 코드는 인라인 백틱 또는 백틱 블록으로 정확히 감쌀 것
  • 표준 번호 · 불릿 리스트를 필요에 따라 사용

GPT-5 Prompting Guide developers.openai.com

  • 코딩 워크플로우 핵심 권고사항 중 하나가 "Markdown standards"
  • 파일 경로 · 함수명 · 클래스를 백틱으로 감쌀 것
  • 코드 펜스 · 리스트 · 테이블을 의미론적으로 올바르게 출력
  • 이는 단순한 가독성이 아니라 모델 출력 품질 자체의 변수

1.4 MCP · 개방형 컨텍스트 프로토콜

MCP ― AI 에이전트의 USB-C 포트

Model Context Protocol은 AI 에이전트가 외부 시스템(Slack · Jira · Drive · 사내 DB)에 접근하기 위한 개방형 표준이다. 도구 정의와 결과 교환 모두 JSON · Markdown 평문 구조를 사용한다.

핵심 사양 code.claude.com

  • 도구 정의 · 결과 교환 모두 평문 구조 (JSON / Markdown)
  • 벤더 중립 ― Anthropic이 주도했으나 사양 자체는 open
  • 외부 시스템 연동을 각 IDE가 따로 구현할 필요 없음
  • 2025년 출시 후 주요 IDE가 빠르게 채택

설정 공유 범위 cross-tool

  • Claude Code ― MCP 네이티브 지원, 표준 정의 주체
  • Cursor ― CLAUDE.md / Skills / MCP 설정 그대로 사용
  • Windsurf, JetBrains 확장도 동일 설정 공유
  • 한 번 정의한 MCP 도구가 여러 IDE에서 즉시 작동

1.4 Markdown 인입 파이프라인

RAG 인입 ― 모든 길이 Markdown으로 통한다

RAG 인입 표준은 사실상 정해졌다. PDF · Office · HTML 등 무엇이든 AI가 읽기 좋은 Markdown으로 일원화하는 도구가 빅테크에서 잇따라 출시되고 있다.

MarkItDown Microsoft Research

  • AutoGen 멀티에이전트 프레임워크의 표준 데이터 파이프라인
  • PDF · Office · HTML 등 다양한 파일 → AI가 읽기 좋은 Markdown으로 일원화
  • "MS가 자사 Office 포맷보다 Markdown을 우선했다"는 자기 인정 시그널

Docling IBM Research

  • 오픈소스 PDF · Office → Markdown / JSON 컨버터
  • 표 · 멀티컬럼 · 헤딩 인식 품질에 집중
  • "결과물은 Markdown"이 기본 출력

LlamaIndex RAG Framework

  • MarkdownNodeParser · MarkdownElementNodeParser
  • Markdown 헤더 기반 청크 분할을 표준 라이브러리에 포함
  • 의미적 계층이 자연스러운 청크 경계 → 검색 정확도 향상

왜 RAG 품질의 가장 큰 변수가 "좋은 파싱"인가

  • 표가 깨지면 숫자가 단락에 섞이고, 산술적 답변이 망가진다
  • 헤딩이 없으면 청크 경계가 의미 단위에서 벗어난다 ― 검색이 엉뚱한 곳을 잡는다
  • 멀티 컬럼이 깨지면 읽기 순서가 뒤섞여 컨텍스트가 의미를 잃는다
  • → 좋은 파싱(=Markdown으로 깨끗하게 변환)이 RAG 품질의 가장 큰 변수 (Firecrawl 2026 PDF 파서 비교)

1.5 표 · 스프레드시트의 한계

Excel은 LLM 입력에서 "노이즈"로 작용한다

정교한 .xlsx 시트는 사람에게는 정보지만, LLM에는 다중 시트 · 수식 · 매크로 · 서식이 모두 해석 비용이다. 실측 사례가 그 격차를 보여준다.

2일 → 20분
핀테크 / 재무 분야 변동 분석 작업 시간 단축
같은 분석을 Excel로 줬을 때는 2일이 걸렸지만, CSV로 변환해 LLM에 입력한 뒤에는 20분으로 단축. 144배 가속. — Stephen Smith, 2026

Excel이 LLM에 어려운 이유

  • 다중 시트 ― 시트 간 참조 / 명명 범위를 모델이 추적할 수 없다
  • 수식 ― 셀 값이 식이라 의미 단위가 깨진다
  • 매크로 / VBA ― 비주얼은 같아도 동작이 숨어 있다
  • 조건부 서식 · 병합 셀 ― 표 구조가 비정형으로 보인다
  • 200,000행을 넘어가면 Excel 자체 성능도 급격히 저하 (KaaShiv)

인용 ― Stephen Smith

  • "LLM이 마치 한 가지 언어만 유창하게 구사하는 분석가라고 생각하라. CSV를 건네면 깨끗한 행과 열을 본다. Excel을 건네면 그 안에서 의미를 추측해야 한다."
  • — Stephen Smith, Why Your Finance Team's Excel Files Are Sabotaging Your AI Success, 2026

1.5 포맷별 트레이드오프

CSV가 항상 정답은 아니다 ― 토큰 vs 정확도의 함수

11가지 포맷(CSV · TSV · JSONL · Markdown 테이블 · YAML 등) 비교 실험은 정답이 데이터 모양과 작업 유형에 따라 달라진다는 점을 분명히 했다.

포맷 정확도 토큰 사용 적합한 상황 한국어 / 금융 적합성
Markdown 테이블key-value 스타일 60.7% 많음 구조와 정확도가 중요한 작업, 소규모 표 OK ― 한국어 헤더에도 안전
CSVcomma-separated 44.3% 적음 대량 데이터, 도구 호환성 우선 주의 ― 값 안 쉼표 이스케이프 오류 가능
TSVtab-separated 유사 (CSV±) 적음 값에 쉼표 포함된 데이터(한글 문장 · 금액) 최적 ― AI Hub 일부도 TSV 배포
JSON / JSONLstructured 높음 많음 중첩 구조 · 반정형 데이터 · API 통합 OK ― UTF-8 안전
Parquetcolumnar binary N/A (직접 입력 X) 압축 2~10× 분석 저장소, SQL 경유 LLM 사용 최적 ― 대용량 처리
⚖
실용 결론
테이블 데이터는 토큰 · 정확도 트레이드오프에 따라 CSV / TSV · Markdown 테이블 · JSON · Parquet+SQL을 선택한다. 한국어 · 금융 데이터는 TSV가 가장 안전한 첫 선택지.

1.5 권장 파이프라인

Excel을 통째로 던지지 말 것 ― SQL 경유 패턴

거대한 다탭 XLSX를 LLM에 직접 넣으면 토큰 폭발 · 헤더 오해석 · 일관된 산술 오류가 발생한다. 업계 권장 패턴은 4단계로 정리된다.

STEP 1
사용자 편집
XLSX · Google Sheets ― 사람의 즉석 분석 · 차트 · 서식은 여기서 그대로
STEP 2
정규화
CSV / TSV / Parquet으로 변환 ― 시스템 저장 · AI 입력 표준
STEP 3
SQL 엔진에 적재
SQLite · DuckDB(또는 BigQuery · Snowflake)로 적재해 컬럼 인덱스 활용
STEP 4
LLM이 SQL 생성·실행
필요한 컬럼·행만 가져와 토큰 폭발과 산술 오류를 모두 회피

Microsoft Fabric ― 자기 인정 시그널

  • Microsoft Fabric이 "Excel → Delta(Parquet)" 변환을 별도 기능으로 정식 출시
  • 패턴: raw 데이터는 받되, 내부 저장은 Parquet
  • MS 자체가 .xlsx의 한계를 운영 단에서 인정한 사례

Parquet의 매력

  • 저장 공간 2~10배 감소 (1GB CSV → 약 200MB)
  • 분석 쿼리 10~100배 가속 (컬럼 프루닝)
  • BigQuery · Snowflake · DuckDB 모두 1급 입력

2.1 기존 워크플로우 — 7단계

기존 방식 ― Excel로 분석, Word/PDF 짜깁기, PPT 배포

대부분의 사무직이 보고서 한 편을 만드는 방식. 각 단계가 다른 도구·다른 포맷을 사용하며, 각 단계마다 정보가 한 번씩 끊긴다.

1 자료 수집 Outlook · 파일서버 이메일 첨부 PDF · Word를 폴더에 모음
2 자료 읽기 Word · Adobe Reader PDF 200+ 페이지 직접 읽고 형광펜
3 데이터 분석 Excel · VLOOKUP · 피벗 13개 분기 시트를 수동 결합 · 수식 디버깅
4 차트 생성 Excel 차트 → 캡처 엑셀 차트를 그림 파일로 export
5 슬라이드 작성 PowerPoint Word에서 본문 복사 · 차트 그림 붙임
6 디자인 PowerPoint 수동 폰트 · 정렬 · 색 · 위치 · 페이지 번호
7 배포 Outlook 첨부 PPT + 별첨 폴더(원본 30+ 파일) 압축

이 흐름에서 사용자가 들이는 시간

  • 2 · 3 단계 (읽기 + 분석) — 보고서 분량의 60%를 차지
  • 5 · 6 단계 (PPT 작성 + 디자인) — 25%가 디자인에 소비
  • 분석 결과가 바뀌면 4 · 5 · 6 단계를 처음부터 다시
  • 분기 1회당 1.5 ~ 2일 소요 (실측)

각 단계 사이의 정보 손실

  • 2 → 3: Word 본문 → Excel 수치 수동 옮김
  • 3 → 4: 분석 결과 → 차트 그림 스크린샷
  • 4 → 5: 차트 그림 → PPT 붙여넣기 (해상도 손실)
  • 5 → 6: 본문 의미 → 슬라이드 디자인 축약

2.2 기존 방식의 5가지 한계

왜 이 방식이 AI 시대에 맞지 않는가

한 번 만든 보고서가 다시 사용되지 않는다. 자료가 검색되지 않는다. AI가 읽을 수 없다. ― 다섯 가지 한계가 누적된다.

① 시간 비용 보고서 한 편당 1.5~2일
  • 자료 읽기 · 형광펜~4h
  • Excel 분석 · 디버깅~4h
  • PPT 디자인 · 정렬~6h
  • 검토 · 수정~2h
② 재사용 불가 다음 분기 시 처음부터 다시
  • Excel 분석 재사용0%
  • 차트 재사용0%
  • 본문 텍스트~20%
  • 구조 · 서식~30%
③ 검색 불가 풀텍스트 검색 안 됨
  • 1년 후 자료 발견~30분
  • 페이지 위치 확인~15분
  • 인용 · 복사~5분
  • AI 컨텍스트 사용불가
④ 서식 충돌 Word → PPT 시 매번 재정리
  • 폰트 · 크기 재지정매번
  • 표 행 · 열 정렬매번

4 비즈니스 로직 계산 · 워크플로우 · 검증 규칙 · 외부 통합
5 도메인 지식 학습 법령 · 규정 · 약관 · 표준 — 개발자가 익혀야 정확한 구현

3.7 도메인 지식을 RAG로 내장하는 5단계

개발자가 학습하지 않는다 — 시스템이 답변한다

어떤 신규 사업이든 도메인 자료(법령 · 규정 · 매뉴얼 · 정책 · 계약서 · FAQ)가 있다. 개발자가 이를 익히는 대신 AI가 인용 답변하도록 RAG로 내장한다. 5단계 통일 패턴이다.

1 도메인 자료 식별 법령 · 규정 · 매뉴얼 · 정책 · 계약서 · FAQ · 사내 위키 등 "개발자가 익혀야 하는 모든 텍스트"를 모은다
2 Markdown 정규화 MarkItDown + 수작업 정제 · PDF/Word/PPTX → .md · 표·헤딩·구조 보존
3 청킹 전략 선택 법령은 조항별(article) · 가이드는 헤더별(markdown) · FAQ는 문단별(paragraph)
4 임베딩 · 인덱싱 1536-dim embedding 생성 → HNSW 벡터 인덱스(pgvector) 적재 · upsert로 변경 자료 즉시 반영
5 권한 격리 visibility(public · role · company) 메타 → match_documents RPC가 SQL 단에서 자동 필터링

어떤 사업에 활용되나 동일 패턴

  • B2B 영업 · 제품 카탈로그 · 가격표 · 경쟁사 비교 · 영업 자료

3.8 End-to-End — 사용자 질문에서 답변까지

10단계로 본 요청 → 응답 흐름

사용자가 도메인 질문을 한다 — 답변에 도달하기까지의 흐름. 이 패턴은 사업이 무엇이든 동일하다. 붉은 단계가 보안 격리 지점이다.

1 USER 사용자가 챗봇 UI에 질문 입력 useChatStore.send(question)
2 CLIENT Firebase ID Token 첨부 → POST /chat (Bearer Auth) ― API 키 직접 노출 금지
3 CF · AUTH verifyAuth — Firebase Admin SDK로 토큰 검증, role · companyId 추출
4 CF · LIMIT rateLimit — uid 기준 분당 호출 제한 (비정상 트래픽 · 비용 폭증 차단)
5 CF · EMBED 질문을 embedding(1536-dim)으로 변환
6 SUPABASE match_documents RPC → HNSW 유사도 검색, visibility 필터 SQL에서 강제
7 CF · AGENT Anthropic messages.stream 호출 ― 검색 결과를 context로, 필요시 tool-call(정산조회 등)
8 CF · TOOL 필요 시 등록된 도메인 함수 호출 — 예: getMonthlySettlement(companyId, …) → DB 조회(companyId 격리 강제) → 결과 반환
9 CF · STREAM SSE 응답 chunk를 클라이언트로 토큰별 푸시 ― 사용자가 답변이 만들어지는 것을 본다
10 CF · AUDIT logChatEvent — 세션 · 메시지 · token usage · 소스 인용을 Supabase에 기록 (감사 추적)
🔒
보안 격리 — 4 단계
단계 2 · 3 · 4 · 10은 모두 보안/감사 격리 지점. 브라우저는 anon/key를 직접 가지지 않고, role/company 필터는 SQL에서 강제하며, 모든 호출은 감사 로그에 남는다. 한 사용자가 다른 회사의 데이터를 절대 볼 수 없다.

  • → 비기술자가 즉시 · 안정적으로 · 모바일에서 쓸 수 있는 유일한 옵션

② 축 ② 클라우드 동기 메모 — Moonwave Memo

Obsidian이 부담스럽다면 — 자체 메모 앱

Obsidian이 너무 무거우면 개인화된 자체 메모 앱이 답이다. Moonwave Memo는 React 19 + Firebase + Tiptap + LangChain으로 만든 자체 개발 예시. 일상 메모 · 회의 노트 · 짧은 아이디어 캡처에 최적 — 본 발표자 본인의 워크플로우.

핵심 기능 (Moonwave Memo 예시)

  • Tiptap WYSIWYG — 마크다운 호환 리치 에디터
  • Firebase Firestore — 실시간 멀티디바이스 동기
  • Dexie (IndexedDB) — 오프라인 캐시 · 빠른 검색
  • LangChain 멀티 AI — Anthropic + OpenAI + Google 동시
  • D3-force 그래프 — 메모 간 관계 시각화
  • Fuse.js 퍼지 검색 · 빠른 노트 발견

패턴의 본질

  • 웹 우선 · URL 하나로 접근 · PWA로 모바일 설치
  • 로컬 우선도 · IndexedDB로 오프라인 동작
  • AI 통합 내장 · 작성 중 자동완성 · 편집 · 요약
  • LangGraph 에이전트 · 메모 자동 정리 · 분류 · 추천
  • 관찰 가능성 · Sentry + LangSmith로 AI 호출 트레이스
⚡
자체 앱의 강점
기존 도구(Notion · Apple Notes)는 AI 통합이 외부 의존이다. 자체 앱은 내 메모 데이터의 흐름을 완전히 통제한다. 어떤 AI 모델을 · 어떻게 호출할지 · 어떤 데이터를 어디로 보낼지 모두 내 결정.

AI 전용 PC — 로컬 LLM과 개인 RAG

메인 PC와 분리된 AI 전용 머신을 두면 작업 흐름이 바뀐다. 클라우드 API 비용 · 프라이버시 · 레이턴시 문제가 한꺼번에 사라지고, 메인 PC는 가벼워진다.

왜 분리해야 하나

  • 프라이버시 — 개인 메모 · 회의록 · 코드가 외부 안 나감
  • 비용 — API 호출 → 전기료 (장기 압도적 절감)
  • 레이턴시 — 서버 왕복 없이 즉시 응답
  • 작업 격리 — 에이전트가 밤새 돌아도 메인 PC 영향 없음
  • 오프라인 가능 — 인터넷 끊겨도 작동
  • 헤드리스 운영 — 모니터 없는 서버로도 충분 · SSH·원격 접근 · 책상 공간 절약

무엇을 실행하나

  • 로컬 LLM · Llama 3 · Mistral · Qwen via Ollama · LM Studio
  • 개인 RAG · Obsidian 볼트 + 메모 → 로컬 벡터 인덱스
  • 에이전트 작업 · Antigravity · Claude CLI · 자동 PR · 리팩터
  • 배치 작업 · 메일 분류 · 회의록 요약 · 자료 정리 (밤새)
  • 이미지 · 음성 · Stable Diffusion · Whisper · TTS
